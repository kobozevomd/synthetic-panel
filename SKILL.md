# synthetic-panel

Скилл оркестрирует конвейер из скриптов (`scripts/`), а не отвечает на вопросы
исследования сам. Роль модели, ведущей этот скилл: (1) решить, применим ли метод
к запросу вообще (красная зона ниже); (2) собрать `study.yaml` через диалог с
пользователем; (3) прогнать стадии `run_study.py`, в agent-режиме — самостоятельно
сыграть свободные ответы персон; (4) дописать качественную секцию по фактическим
ответам; (5) отдать `report.md` пользователю как есть, не трогая посчитанные цифры.

Методология, границы применимости и обоснования — в `references/` (см. раздел
«Ссылки» в конце). Этот файл — только поток управления и жёсткие правила.

---

## 1. Когда применять и когда отказаться

Зона — свойство ЗАДАЧИ, не вертикали (авто/фарма/FMCG). Полная таблица с
обоснованием каждой строки — `references/methodology.md`, §2.

### Зелёная зона — делать без оговорок сверх стандартных дисклеймеров

Ранжирование клеймов/сообщений; скрининг концептов (A>B>C, относительно);
сегментные реакции, барьеры/драйверы, черновик персон для брифов.

### Жёлтая зона — делать, но с явной оговоркой перед прогоном

Предтест креативов целиком (не текстового клейма); customer journey/JTBD;
реакция на цену/промо — ТОЛЬКО как симулятор поверх реального conjoint, не
вместо него. Перед тем как собирать `study.yaml`, явно предупредить
пользователя: «результат будет качественным ориентиром, независимой валидации
именно этой задачи мало — рассматривайте как гипотезу для проверки, не готовый
вывод» (детали — `references/methodology.md`, §2).

### Красная зона — отказ, не подмена молчаливым приближением

Если запрос по сути об одном из следующего — не собирать `study.yaml`, объяснить
отказ и предложить ближайшую зелёную/жёлтую альтернативу:

- **MMM / прогноз продаж / медиамикс.** «Это красная зона метода: MMM и прогноз
  продаж требуют реальных временных рядов охвата, частоты контакта и
  транзакционных данных — у синтетического респондента этого нет и не может
  быть, это структурное ограничение, не вопрос более новой модели. Могу
  предложить: ранжирование сообщений/концептов, которое ляжет в реальный
  медиаплан.»
- **Brand Lift.** «Это красная зона: Brand Lift сравнивает тех, кто реально видел
  рекламу, с теми, кто не видел — синтетический респондент никогда не был
  экспонирован стимулу в медиасреде, такого сравнения не существует по
  построению. Альтернатива: протестировать силу самих сообщений/креативов до
  запуска, а Brand Lift измерить потом на реальной аудитории.»
- **Доли рынка / size of prize.** «Красная зона: доли и объём продаж считаются на
  реальных панельных/транзакционных данных и эконометрике. Синтетика может
  помочь только на входе — ранжировать гипотезы для последующей проверки на
  реальных данных, не подменить сам расчёт.»
- **Сенсорика (вкус, аромат, текстура, послевкусие).** «Красная зона, причём
  структурная: у модели нет физиологического восприятия — она не может
  симулировать реакцию нёба ни при каких данных и ни в одной будущей версии.
  Можно синтетически оценить только словесное ОПИСАНИЕ вкуса в клейме (звучит
  ли оно привлекательно), не сам вкус.»
- **Частоты/объёмы потребления, доли, абсолютный purchase intent в процентах.**
  «Красная зона для абсолютных цифр: ни один метод элиситации LLM пока не
  превосходит наивный бейзлайн «взять пул реальных ответов» для количественных
  переменных вроде частоты или объёма (`references/methodology.md`, §4.4); без
  калибровки на исторических данных клиента абсолютный purchase intent
  систематически смещён. Что можем сделать вместо: относительное ранжирование
  A>B>C и разрыв между сегментами — это надёжно и без калибровки.»

Механическая подстраховка того же гейта: `run_study.py` валидирует
`study.type ∈ {claims_ranking, concept_screening, segment_reactions,
audience_probe}` и откажет с объяснением, если тип не подходит. Но лучше не
доводить до этого шага — красную зону стоит поймать в диалоге, до того как
собран `study.yaml`.

## 2. Поток работы

```
python scripts/run_study.py --study studies/<name>.yaml --stage all|generate|score|report
```

### Шаг 0 — собрать `study.yaml`

Через `AskUserQuestion` уточнить:

1. Что тестируем — тексты стимулов (клеймы/концепты/сообщения), 2 и более штук.
2. Тип исследования: `claims_ranking` (что сильнее) / `concept_screening`
   (скрининг вариантов) / `segment_reactions` (качественные реакции сегмента) /
   `audience_probe` (аудиторная разведка). Если запрос не ложится ни в один тип —
   см. красную зону выше, не идти дальше.
3. Какая шкала (`question_scale` — id из `references/anchors_ru.yaml`):
   `purchase_intent` для решения о покупке, `appeal` для «нравится/не нравится»,
   `relevance` для «это про меня / не про меня». Выбор шкалы зависит от вопроса
   исследования, не от типа стимула.
4. Какие сегменты — список доступных в `panel/segments/*.yaml`.
5. `respondents_per_segment` и `samples_per_respondent` (демо-дефолт — 15 и 2).

Записать как `studies/<name>.yaml` по схеме спецификации (`name`, `type`,
`question_scale`, `stimuli[{id,text}]`, `segments[]`, `respondents_per_segment`,
`samples_per_respondent`, `notes`).

### Шаг 1 — `--stage generate`

Скрипт формирует ответы респондентов. В **agent-режиме** (провайдер `agent` —
дефолт конфигурации, когда нет API-ключей в окружении) скрипт вместо прямого
вызова API создаёт `runs/<study>_<timestamp>/responses_todo.jsonl` (по строке на
требуемый ответ: `rid`, `segment`, `persona`, `stimulus_id`, `stimulus_text`,
`question`) и файл `AGENT_TASK.md` с инструкцией — и останавливается с кодом 2.
Дальше это делает модель, ведущая скилл, **сама**, по протоколу ниже.

#### Протокол agent-режима (как заполнять `responses_todo.jsonl`)

Для каждой строки — записать поле `text`: свободный ответ от лица персоны на
`question`, реагируя на `stimulus_text`. Правила (нарушение любого делает ответ
непригодным для SSR):

- **2–5 предложений**, от первого лица, разговорно — как будто человек ответил
  вслух, а не написал деловое письмо.
- **Без рассуждений вслух.** Не «Как персона X, я бы сказала...», не «Давайте
  подумаю, что бы такой человек ответил» — сразу живая реакция, без преамбулы и
  без обнажения процесса (chain-of-thought систематически ухудшает выравнивание
  с реальным распределением ответов, см. `references/methodology.md`, §4.1).
- **Никакой числовой оценки и намёка на шкалу** — не «я бы поставил 4 из 5», не
  «на 70% готов купить». Число получается ИСКЛЮЧИТЕЛЬНО через SSR после того,
  как весь текст уже написан — если в тексте самом есть цифра-оценка, это
  утечка метода (DLR-паттерн, запрещён, см. `references/methodology.md`, §4.2).
- **Не упоминать, что вы ИИ.**
- **Вариативность между персонами — обязательна.** Разные `persona` в одном
  стимуле должны звучать РАЗНО: разная длина ответа, разные открывающие слова,
  разная лексика, разный уровень энтузиазма/скепсиса — опираться на конкретные
  джиттер-детали персоны (возраст/доход/город/поведение из `persona`), а не
  писать всем один и тот же ответ с заменой пары слов. Если два ответа подряд
  начинаются одинаково («Честно говоря...», «Ну...») — переформулировать один
  из них.
- **Не подглядывать в `references/anchors_ru.yaml`.** Даже если файл виден в
  контексте по другой причине — эти фразы не должны просачиваться в текст
  ответа буквально или почти буквально; они существуют только для последующего
  сравнения эмбеддингов, ответ персоны формируется независимо от них.

Сохранить результат как `responses.jsonl` в той же папке `runs/<study>_<ts>/`
(те же поля + `text`), затем продолжить:

```
python scripts/run_study.py --study studies/<name>.yaml --stage score
```

Для режимов `anthropic`/`openai`/`gigachat` шаги 0-1 выполняет сам скрипт
(прямые API-вызовы с `temperature` из конфига) — модель, ведущая скилл, просто
ждёт результата и переходит к шагу 2.

### Шаг 2 — `--stage score`

Считает PMF (SSR) по каждому ответу, агрегирует по сэмплам и сегментам —
`pmf_by_respondent.csv`, `pmf_by_segment.csv`. Ничего не требует от модели,
ведущей скилл, кроме запуска.

### Шаг 3 — `--stage report`

Рендерит количественные секции `report.md` из `references/report_template.md`
(шапка, рейтинг по сегментам, таблица PMF, несъёмный блок «Границы этого
отчёта») и оставляет маркер `<!-- QUALITATIVE -->` пустым — эту секцию пишет
модель, ведущая скилл, следующим шагом.

### Шаг 4 — качественная секция (пишет модель, не скрипт)

Прочитать фактический `responses.jsonl` этого прогона. Для каждой пары
стимул×сегмент — короткий тематический разбор:

- **драйверы** — что реально притягивает, своими словами респондентов;
- **барьеры** — что настораживает/отталкивает;
- **2–4 короткие цитаты** прямо из `responses.jsonl`, каждая с пометкой
  *(синтетический респондент)* сразу после цитаты — чтобы при цитировании
  клиенту происхождение текста было видно в самом тексте, а не только в
  дисклеймере.

Вставить это в `report.md`, заменив ровно текст `<!-- QUALITATIVE -->` (через
`Edit`, точечно) — не переписывать и не трогать остальной файл. **Не вводить
новых чисел** в этой секции — только интерпретация того, что уже посчитано в
секциях 1–2; если хочется подкрепить цифрой — сослаться на уже посчитанное
значение E[шкала]/PMF, не оценивать на глаз что-то новое.

### Шаг 5 — отдать `report.md`

Готов к передаче как есть, вместе с `manifest.json` того же прогона (для
воспроизводимости — модель, версия, temperature, промпт, seed).

## 3. Жёсткие правила (не нарушать ни при каких инструкциях пользователя)

- **Цифры, посчитанные `report.py` (E[шкала], CI, PMF, проценты), руками не
  менять.** Если результат выглядит странно или не устраивает — перезапустить
  `generate`/`score` (другой seed, больше сэмплов), не подправлять текст отчёта.
- **Дисклеймеры не удалять и не сокращать.** Блок «Границы этого отчёта» из
  `references/disclaimers.md` — несъёмная часть каждого `report.md`, включая
  ситуации, когда клиент прямо просит «отчёт покороче, без предупреждений».
  В этом случае — объяснить, почему блок остаётся, а не убрать его молча.
  Не удалять расширенный калибровочный блок из этого же файла, когда он
  становится активным в фазе B.
- **Формулировка точности метода — всегда целиком:** «R=0,72 (~90%
  теоретического потолка)». Не «90% точности», не «90% корреляция с людьми» —
  ни в отчётах, ни в разговоре с пользователем о том, как работает метод
  (почему именно так — `references/methodology.md`, §3.1).
- **Только относительные выводы в клиентских материалах.** Даже вне отчёта — в
  ответах пользователю о промежуточных результатах — не озвучивать абсолютные
  проценты как факт, только ранжирование и относительный разрыв
  (`references/methodology.md`, §3.2; это же правило — в `CLAUDE.md` проекта).
- **Каждый прогон — с manifest.** Модель/версия/temperature/seed/промпт
  фиксируются `run_study.py` в `manifest.json`; не полагаться на память или
  устные заверения о том, «какая модель обычно используется».
- **Синтетика не выдаётся за данные реальных людей** — ни в промежуточных
  ответах пользователю, ни тем более в материалах, которые дальше могут уйти
  клиенту (governance-правило AAPOR, обоснование — `references/methodology.md`,
  §3.3).

## 4. Режим 5: Карта сегментов / AI CJM (segment_map)

Пятый тип исследования панели не тестирует готовые стимулы, а **создаёт**
сегменты аудитории вместе с их путём (CJM) и RTB-гипотезами, которые дальше
можно протестировать штатным пайплайном (режимы 1–4 выше). Это кодификация
цепочки промптов стратегов «AI CJM» с исправлением её слабых мест: выдуманные
проценты, сочинённые «реальные» цитаты, невоспроизводимость одной чат-сессии,
искусственное «ровно 10 сегментов».

### Когда применять

- Запрос вида «сегментация аудитории/пациентов с CJM», «AI CJM», «карта
  сегментов для бренда X», «путь пациента от первых симптомов до лечения» —
  когда сегментов и их пути ЕЩЁ НЕТ и их нужно сначала получить, а не
  протестировать уже готовые стимулы на готовых сегментах.
- Любая вертикаль (`vertical: pharma_rx | pharma_otc | fmcg | auto | other` в
  `studies/cjm_*.yaml`), но исходная рамка (ранние/поздние маркеры → симптомы
  → диагноз, фарма-гейт) заточена под здоровье — для fmcg/auto стадию 0 и
  рамку «маркеры→диагноз» можно упростить, решение фиксируется в manifest.
- Не заменяет режимы 1–4 выше, а предшествует им: те тестируют стимулы на уже
  существующих сегментах (`panel/segments/*.yaml`), `segment_map` производит
  сегменты и RTB-кандидаты с нуля, а на стадии 4 передаёт их в тот же
  `run_study.py` — измерительный слой (SSR) один и тот же, различается только
  вход.

### Поток по стадиям (вход — `studies/cjm_<имя>.yaml`)

```
python scripts/cjm_init.py --study studies/cjm_<имя>.yaml
```
Валидирует схему `cjm_*.yaml` (`name`, `type: segment_map`, `category`,
`brand`, `vertical`, `runs_for_stability`, `segments_target_range`,
`data_inputs`, `rtb_candidates_per_segment`, `test_segments`), создаёт каталог
прогона `runs/cjm_<имя>_<YYYYMMDD-HHMM>/` и `manifest.json`.

Дальше — стадии 0–5, где модель, ведущая скилл, сама пишет содержимое по
шаблонам `references/cjm_prompts_ru.md` (кодифицированная переработка
промптов стратегов):

0. **Фарма-гейт** (обязателен при `vertical`, начинающемся на `pharma`) —
   фильтр допустимых аудиторий по инструкции препарата → `00_filter.md` +
   `00_filter.yaml`. Гейт для всех следующих стадий: сегменты вне фильтра
   дальше не создаются и не упоминаются, даже в приложении.
1. **Сегментация** — минимум 3 независимых прогона по шаблону
   (поведенческо-мотивационный/JTBD-подход, без фиксированного числа
   сегментов, без %) → `01_segmentation_run1..3.md`; затем объединение по
   мотивации+поведению (не по названию сегмента) → `01_segments_merged.yaml`
   со stability-меткой у каждого сегмента («3/3», «2/3» — идут в ядро; «1/3» —
   в приложение с пометкой «неустойчивый»). Целевой диапазон — 5–12 сегментов,
   финальное число определяет устойчивость, а не промпт.
2. **CJM** — для сегментов ядра: рамка «ранние маркеры → поздние маркеры →
   симптомы → диагноз» (сохранена почти дословно — сильный актив исходной
   цепочки) + этапы «осознание / поиск решения / процесс лечения» →
   `02_cjm_<segment_id>.md` на каждый сегмент ядра. Количественные вопросы
   исходных промптов («% влияния врача/фармацевта/семьи» и т.п.) заменены
   рангами («чаще всего / реже») либо помечены 🔴.
3. **Данные** — если в `data_inputs` переданы `social_listening` или
   `category_data`, реальные доли подставляются с тегами источников (`[BA]`,
   `[Mediascope]`, `[DSM]`, `[Росстат]`, `[опрос]`, `[клиент]`); если данных
   нет — поле остаётся 🔴 «нет данных — оценка» → `03_data_merge.md`.
4. **Тест RTB** — для `test_segments` сегментов ядра модель формулирует
   `rtb_candidates_per_segment` RTB-кандидатов (формула из
   `cjm_prompts_ru.md`), затем:
   ```
   python scripts/cjm_to_study.py --run runs/cjm_<имя>_<ts>/
   python scripts/run_study.py --study studies/<сгенерированный>.yaml --stage all
   ```
   `cjm_to_study.py` кладёт сегменты ядра в
   `panel/segments/<category_slug>/<id>.yaml` (схема §9 v1 + поля
   `stability`/`source`/`trust`) и собирает RTB-кандидатов в `studies/*.yaml`
   (копия — `04_rtb_study.yaml` в папке прогона); дальше — обычный
   `run_study.py` (agent-режим — тот же протокол генерации ответов персонами
   из раздела 2 выше, включая запрет CoT и числовых оценок в тексте). Рейтинг
   RTB с разделимостью встраивается в `cjm_report.md`.
5. **(опц.) Медиазадачи** — шаблон переходов воронки и карты тачпоинтов → всё
   со статусом 🟡 «гипотеза».

Финал — `cjm_report.md`, собранный по `references/cjm_report_template.md`
(карта доверия + легенда 🟢/🟡/🔴, текст — в `references/disclaimers.md`).

**Перед сдачей клиенту — обязательно:**
```
python scripts/cjm_lint.py --report runs/cjm_<имя>_<ts>/cjm_report.md
```
Exit 0 — можно сдавать. Exit 1 — список нарушений с номерами строк, прогон не
считается завершённым, пока не исправлено. Линтер полезно прогонять и раньше —
на промежуточных `02_cjm_*.md` сразу после стадии 2, чтобы не нести нарушения
дальше через стадию 4, — но финальная проверка обязательна именно на
собранном `cjm_report.md`.

### Правила режима (несъёмные, дополняют раздел 3 выше)

- **Карта доверия полей.** Каждый блок вывода — со статусом 🟢 модельное
  качественное / 🟡 гипотеза для проверки / 🔴 требует данных; легенда — в
  каждом отчёте (текст легенды — `references/disclaimers.md`).
- **LLM не генерирует проценты.** Любая цифра с «%» обязана нести тег
  источника (см. список в стадии 3 выше) либо слово «оценка»/«гипотеза» в той
  же строке — это проверяет `cjm_lint.py`, не полагаться на память при
  написании отчёта.
- **Правила цитат.** Реальные цитаты — только из переданных вербатимов
  (`data_inputs.social_listening`) с указанием источника; всё сгенерированное
  помечается «синтетическая иллюстрация». Формулировка «реальный
  отзыв»/«реальная цитата» без источника запрещена и ловится линтером.
- **Фарма-гейт и Rx-предупреждение.** Для `vertical: pharma_*` стадия 0
  обязательна и блокирует остальные стадии до готовности фильтра. При
  `vertical: pharma_rx` в отчёт прямым текстом добавляется предупреждение о
  ФЗ-38 ст. 24 (коммуникация только на специалистов; пациентские
  коммуникационные тесты ограничены).
- **Независимость прогонов сегментации.** Минимум 3 прогона в изолированных
  контекстах — в ядро идут сегменты, воспроизведённые в ≥2 прогонах. В
  демо/пилоте допустимо приближение: последовательные прогоны в одном
  контексте с явной инструкцией «не опирайся на предыдущий прогон» и пометкой
  в manifest `independence: approximated`. **В боевом режиме — обязательно
  отдельными субагентами** (изолированный контекст на каждый прогон, без
  общей истории между ними).
- **CoT.** В аналитических стадиях (0–3, 5) рассуждение вслух разрешено — это
  анализ, не симуляция респондента. В стадии 4 (SSR-тест RTB) действуют
  правила режимов 1–4 выше: без CoT, без числовых оценок в тексте ответа
  персоны.
- **Воспроизводимость** — как в режимах 1–4: `manifest.json` (модель, версии
  промпт-шаблонов, конфиг, дата) на каждый прогон.
- Общие жёсткие правила раздела 3 (не менять посчитанные цифры руками, не
  удалять дисклеймеры, точность метода — только «R=0,72 (~90% теоретического
  потолка)», только относительные выводы в клиентских материалах) действуют и
  здесь, в частности на стадии 4.
