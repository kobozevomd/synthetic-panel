# Заметки к scripts/render_client.py ([B4], spec_synthetic-panel_v1.3.md §2.2)

Короткий документ для F1-ревью и следующих итераций — не инструкция пользователю
(она не нужна: `python scripts/render_client.py --report <path>` самоочевиден),
а фиксация решений, которые не очевидны из самого кода.

## 1. Парсинг — по заголовкам и сигнатурам столбцов, не по номерам строк

`render_client.py` не знает фиксированной схемы report.md/cjm_report.md. Пять
разделов распознаются по НОРМАЛИЗОВАННОМУ тексту заголовка (без нумерации,
эмодзи-маркеров, ё/е): «Главное», «Что с этим делать»/«Что делать», «Границы
этого отчёта» (в т.ч. с любым суффиксом — «...карта сегментов / AI CJM»,
«...(общие)» — склеенный вариант v1.3 §2.1.4 попадёт в ОДИН такой раздел с
подзаголовками внутри, старый вариант из нескольких раздельных H2 — в
несколько; оба варианта корректно объединяются в финальный блок «Границы»),
«Легенда карты доверия», «Паспорт методологии». Все пять — ОПЦИОНАЛЬНЫ: старые
отчёты (v1/v1.2, оба пилота на диске сейчас) их не имеют — рендер просто
пропускает соответствующий блок, не падает и не выдумывает контент. Любой
раздел вне этих пяти рендерится generic-веткой в исходном порядке — ничего не
пропадает молча.

Таблицы-рейтинги находятся ГЛОБАЛЬНО по сигнатуре столбцов (`is_ranking_table`:
"Место" + один из "E[.../Отделим.../Разделим.../Устойчив..."), не по разделу —
одинаково находит их в `## 1. Рейтинг...` (report.md) и в `### 6.2 Результат
SSR-теста` (cjm_report.md). Названия столбцов ищутся подстрокой/word-boundary-
регэкспом (`_find_col`), не точным текстом.

## 2. Важная находка: v1.3 уже меняет форму report.md ПАРАЛЛЕЛЬНО этой работе

При старте этой сборки `references/report_template.md`/`scripts/report.py`
были ещё в форме v1.2 (тех-шапка, единая таблица PMF, двухуровневая
разделимость). К концу сборки [B1]/[B3] обновили их до v1.3 (см. spec §2.1,
§1.3.3) — обнаружено при регрессионном прогоне `scripts/test_*.py`. Это
означает, что секция 1 report.md БОЛЬШЕ НЕ содержит E[шкала]/CI/PMF — только
3-колоночную таблицу (Место/Стимул/ярлык разделимости), а сырые числа переехали
в новый раздел `## Приложение` (технический паспорт + `### Полная статистика по
сегментам` — таблица-рейтинг НА СЕГМЕНТ с полными E/95% CI/PMF/P(A>B)/Ярлык).

`render_client.py` уже учитывает это:
- `is_ranking_table`/`extract_rank_rows` понимают колонку "Устойчивость разрыва
  от следующего" (v1.3), не только "Отделимость от следующего" (v1/v1.2).
- `classify_separability` понимает форму последней строки ранжирования "—
  (Топ-N воспроизведён в X из Y проверок…)" (не голое тире) — распознаёт её как
  tier `none`, сохраняя пояснение в скобках для отображения.
- `classify_heading` распознаёт «Приложение» как отдельный вид
  (`appendix_section`) — его содержимое уходит в тот же `<details class=
  "appendix">`, что и авто-найденные PMF-таблицы старых отчётов. Таблицы ВНУТРИ
  такого раздела рендерятся `render_body(..., cardify_rankings=False)` —
  ОБЫЧНЫМИ HTML-таблицами, не карточками: карточка намеренно прячет CI/P(A>B),
  а раздел «Приложение» существует именно ради «сырых цифр для собственного
  анализа» (буквально из текста report_template.md) — карточка там была бы
  контрпродуктивна.
- `build_pmf_lookup_from_document` — мост между секцией 1 (в v1.3 без своей
  колонки PMF) и «Приложением» (где PMF есть, но ASCII, под тем же заголовком
  «Сегмент: X», НЕ колонкой таблицы, как раньше): без этого моста бары секции 1
  были бы всегда «нет данных PMF в отчёте» для любого report.md, собранного
  новым report.py — честно, но обедняло бы ключевую фичу задания (§2.2:
  «горизонтальные SVG-бары распределений»).

Всё это проверено фикстурой `CLEAN_REPORT_MD_V13_SHAPE` в test_render_client.py
(класс `TestForwardCompatibleV13ReportShape`), списанной с фактического
`references/report_template.md`/вывода `report.py` на момент написания
(`references/cjm_report_template.md` в итоге обновлён до той же формы — см.
п.2.1 ниже). В процессе этой сборки `scripts/test_report.py` у [B1] был короткое
время красным (3 теста — переименование функции без правки теста), к моменту
сдачи снова зелёный — пайплайн v1.3 у [B1] на глазах стабилизировался прямо во
время этой работы; тем не менее форма шаблонов ещё может измениться до
финальной интеграции [F1]/[F2] — если это случится, следующие точки в
render_client.py нужно свериться первыми: `is_ranking_table`, `extract_rank_rows`,
`classify_heading`, `build_pmf_lookup_from_document`, `classify_separability`.

## 2.1. Ещё одна находка (последняя сверка перед сдачей): cjm_report_template.md
тоже уже обновлён до v1.3, причём иначе, чем можно было предположить

`references/cjm_report_template.md` на момент сдачи ТОЖЕ переведён на v1.3
(«Главное»/«Паспорт методологии»/«Что с этим делать»/единое «Границы этого
отчёта» с H3-подразделом/«Приложение» — все распознаются классификатором так
же, как в report_template.md, включая новый `appendix_section`). Раздел 6.2
(«Результат SSR-теста») теперь ТОЖЕ показывает только 3-колоночную таблицу
(Место/RTB-кандидат/ярлык устойчивости) — но, в отличие от report.md, его
E[шкала]/CI/PMF НЕ переезжают в приложение ЭТОГО cjm_report.md: они остаются в
приложении ОТДЕЛЬНОГО файла `runs/<rtb_study_name>_<ts>/report.md` (под-прогон
стадии 4), на который cjm_report.md только ссылается путём, не встраивает
содержимое. Значит для НОВОГО (v1.3) cjm_report.md мост
`build_pmf_lookup_from_document` не найдёт PMF нигде В ЭТОМ документе — бары
секции 6.2 честно покажут «нет данных PMF в отчёте» (не будет ошибкой,
это корректный результат для однофайлового рендера — см. рамку задания «из
ПОЛНОГО report.md/cjm_report.md», один файл, не агрегация по под-прогонам).
Сегодняшний пилот `cjm_hairloss_demo_20260710-0017/cjm_report.md` (v1.2-форма,
ASCII PMF инлайн в разделе 6.2) бары показывает — регрессии для СУЩЕСТВУЮЩЕГО
отчёта нет, ограничение проявится только когда этот cjm_report.md пересоберут
под v1.3. Если это станет проблемой — самый дешёвый фикс НЕ в этой зоне: агенту,
собирающему cjm_report.md вручную, копировать ASCII-бар из под-прогона в секцию
6.2, как раньше (`{{ASCII_BAR}}`-подобный столбец), а не только ярлык.

## 3. Открытая проблема НЕ этой зоны: cjm_lint отказывает report.md (не только cjm_report.md)

Хвостовой гейт §2.2(а) буквально требует «рендер отказывается работать, если
отчёт не проходит cjm_lint» — `render_client.py` честно вызывает
`cjm_lint.lint_file()` как есть (не дублирует и не ослабляет его правила).

Фактическое положение дел на момент сдачи: `cjm_report.md` (оба тестовых
пилота из формулировки задания) — единственный из двух реальных отчётов,
который ПРОХОДИТ cjm_lint (exit 0, как и в его собственном manifest.json).
`runs/biotinal_claims_20260717-1109/report.md` — НЕ проходит (11 нарушений),
по двум причинам, обе вне зоны [B4]:

1. **Правило 3** (карта доверия 🟢/🟡/🔴 + легенда) требует их в КАЖДОМ H2-
   разделе документа — но report.md (claims_ranking/concept_screening/...)
   никогда, по спецификации v1/v1.2, не должен был иметь эту карту доверия —
   это концепция режима `segment_map` (cjm_report.md). Складывается ощущение,
   что правило 3 cjm_lint.py применяется к report.md, для которого оно
   концептуально не предназначалось.
2. **Правила 1/4** ложно матчят «95%» (в фразе «95%-му бутстреп-ДИ», «95% CI» —
   заголовок таблицы) и «Brand Lift»/«прогноз продаж» (внутри ДОСЛОВНО
   скопированного блока `references/disclaimers.md`, где эти слова легитимно
   объясняют, что НЕ оценивалось). Причина — `report.py::render_report()`
   вставляет содержимое блока `DISCLAIMER_BLOCK_START/END`, но НЕ сами
   строки-маркеры (см. `_extract_block`/`_splice_block` в report.py) — поэтому
   маскировка `cjm_lint.mask_reference_blocks()` (которая ищет ИМЕННО эти
   строки-маркеры) не срабатывает на готовом report.md, хотя срабатывает на
   cjm_report.md (там агент, собирающий отчёт вручную по шаблону, переносит
   маркеры дословно — см. `runs/cjm_hairloss_demo_20260710-0017/cjm_report.md`,
   строки с `<!-- DISCLAIMER_BLOCK_START -->`).

Это НЕ дефект `render_client.py` — рендер реализует правило (а) буквально, как
задано, и корректно откажет ИЛИ пропустит любой report.md/cjm_report.md в
зависимости от РЕАЛЬНОГО результата `cjm_lint`, а не от захардкоженного
предположения (см. `test_render_client.py::TestRealPilotReports` — тест сверяет
поведение с фактическим `cjm_lint.lint_file()` результатом на момент прогона,
поэтому не сломается ни от чьей законной правки в другой зоне). Пока это не
починено, CLI-рендер `runs/biotinal_claims_20260717-1109/report.md` отказывает
(корректно, по духу правила) — но САМА ЛОГИКА рендера уже сейчас корректно
строит карточки/бары/дисклеймеры для этого файла (см.
`test_build_html_core_logic_works_on_both_real_reports_bypassing_lint_gate`,
вызывает `build_html()` напрямую, минуя гейт).

Рекомендация (не для этой зоны, для [B1]/[B3]/F1): либо (i) report.py сохраняет
сами строки-маркеры `DISCLAIMER_BLOCK_START/END` в финальном report.md (тогда
маскировка cjm_lint заработает на report.md так же, как на cjm_report.md), либо
(ii) правило 3 cjm_lint.py применяется только к отчётам режима `segment_map`/
`competitive_positioning` (например, по наличию строки «AI CJM»/«segment_map» в
шапке, или отдельным CLI-флагом), а не универсально к каждому .md. Как только
один из этих двух пунктов починен, `render_client.py` подхватит это без единой
правки — гейт уже реализован через честный вызов `cjm_lint.lint_file()`.

## 4. Прочие решения, не покрытые прямо в докстрингах кода

- **Мод-бейдж** — считается из `manifest.json` (`resolve_mode`), не из
  `{{MODE_BADGE}}` в теле report.md (v1.3 добавляет туда «**Режим прогона:**
  🟡 РАЗВЕДОЧНЫЙ» прямо под H1) — оба источника читают одно и то же поле
  `manifest.json: mode`, поэтому не расходятся; текст в теле отчёта просто не
  парсится отдельно (избыточно). Если поле `mode` отсутствует (старые
  manifest.json, оба текущих пилота) — безопасный дефолт `exploratory` с
  пометкой «не размечен в manifest» у бейджа.
- **controls_failed** ищется по нескольким вероятным путям схемы
  (`resolve_controls_failed`) — на момент старта этой сборки поле нигде не
  встречалось в manifest.json; к моменту сдачи `scripts/test_run_study_controls.py`
  уже существует и активно, но конкретный путь поля в итоговом manifest.json
  не был повторно сверен построчно — при рассинхроне поправить только
  `_dig(...)`-пути в `resolve_controls_failed`.
- **Дисклеймеры**: правило (б) реализовано как пер-пункт сверка (каждый `- **Метка:**
  ...` дисклеймера, как логический блок через переиспользованный
  `cjm_lint.split_into_blocks`, обязан дословно найтись в отрендеренном
  plain-тексте) — не просто сравнение длины строкой, а честная проверка
  содержимого; см. `assert_disclaimers_not_shortened`.
