# Промпт-шаблоны режима «Карта сегментов / AI CJM» (segment_map)

Кодифицированная переработка цепочки промптов стратегов — «Логика работы_AI CJM
Пациенты_17.06.docx» (15 промптов) и «Промпт для анализа сегмента.docx» — под
пайплайн `segment_map` (`spec_synthetic-panel_v1.1_segment_map.md`, `SKILL.md`
§4). Что сохранено почти дословно, что вырезано и что добавлено — обоснование в
`Анализ_AI_CJM_и_интеграция_с_панелью.md` (§1–3) и правила переработки в спеке
§6. Сводная таблица соответствия «промпт стратегов → стадия здесь» — в конце
файла.

**Как этим пользоваться.** Каждая стадия ниже — это промпт-шаблон с
плейсхолдерами `[В КВАДРАТНЫХ СКОБКАХ]`. Модель, ведущая скилл, подставляет
значения из `studies/cjm_<имя>.yaml`, `manifest.json` и артефактов предыдущих
стадий, прогоняет получившийся текст как задание для себя (стадии 0–3, 5 — в
том же или отдельном контексте, стадия 1 — обязательно в изолированных
контекстах для боевого режима, см. ниже), и сохраняет результат по схеме
артефакта, указанной в конце каждого раздела. Схемы, отмеченные «схема
зафиксирована в коде» — это НЕ предложение, а точное соответствие тому, что уже
читают `scripts/cjm_to_study.py` и `scripts/segments_export.py` — расхождение с
ними ломает стадию 4 механически, не только стилистически.

**CoT.** Стадии 0–3 и 5 — аналитическая работа, рассуждение вслух (chain-of-
thought) разрешено и даже полезно. Стадия 4 состоит из двух разных по природе
частей: формулировка RTB-кандидатов (аналитика, CoT разрешён, делает модель,
ведущая CJM-прогон) и последующий SSR-тест этих кандидатов на персонах через
`run_study.py` (симуляция респондента, CoT запрещён категорически, числовая
оценка в тексте ответа запрещена — это существующий протокол v1, не
переопределяется здесь, см. `SKILL.md` §2 и `references/methodology.md` §4.1–4.2).

**Карта доверия (кратко).** 🟢 модельное качественное · 🟡 гипотеза для
проверки · 🔴 требует данных. Полная легенда с формулировками для клиента —
`references/disclaimers.md`, раздел «Легенда карты доверия»; она же переносится
в каждый `cjm_report.md` через `references/cjm_report_template.md`.

**Правила честности, действующие во ВСЕХ промптах ниже (не повторяются в
каждом разделе отдельно):**

- Никаких запросов процента или «коэффициента значимости» модели — ни разу, ни
  в одном шаблоне. Где исходная цепочка стратегов просила долю — здесь либо
  ранг («чаще всего / реже», «сильнее / слабее»), либо статус 🔴 с пометкой
  «оценка», либо тег реального источника, если данные переданы
  (`[BA]` / `[Mediascope]` / `[DSM]` / `[Росстат]` / `[опрос]` / `[клиент]`).
- Число сегментов не задаётся — целевой диапазон 5–12 (`segments_target_range`
  в `cjm_<имя>.yaml`), финальное число определяет устойчивость, а не промпт.
- Цитаты: тег `[BA]` или `[отзывы: <путь к файлу>]` — если дословно взяты из
  переданных вербатимов (`data_inputs.social_listening`); без такого тега
  формулировка помечается «синтетическая иллюстрация» (формулировки вида
  «отзыв пользователя» или «прямая цитата» без source-тега в отчётных
  материалах не используем — и в черновиках тоже, чтобы не переносить эту
  привычку дальше в `cjm_report.md`, где это уже ловит линтер).
- Веб-факты — только с URL и датой обращения в этом же прогоне; без
  подтверждённого в ЭТОМ прогоне URL — пометка «по памяти модели, не
  проверено» и статус 🔴/🟡 в зависимости от того, насколько факт критичен.
  Обещание исходной цепочки «всегда перепроверяй свежие данные в интернете»
  сюда не перенесено — оно неисполнимо честно (см. `Анализ_...md`, слабость №7).
- Фарма-поправка на защитное поведение модели — обязательна в стадии 2 (детали
  там), но общий принцип держать в уме на всех стадиях с пациентской темой:
  нейтральная рамка вопроса («как это обсуждают», не «что вы будете делать»),
  нонкомплаенс и самолечение — норма для описания, не исключение.

## Стадия 0 — Фарма-фильтр (фильтр допустимых аудиторий)

Кодификация промпта 1 стратегов — сохранена почти дословно, это их сильная и
редкая практика (регуляторный фильтр первым шагом, а не последним). **Обязательна
при `vertical: pharma_rx` или `pharma_otc`; для `fmcg`/`auto`/`other` стадия
пропускается** (это фиксирует `cjm_init.py` в плане стадий — см. `SKILL.md`
§4). Гейт для всех следующих стадий: ни один сегмент вне разрешённой аудитории
дальше не создаётся и не упоминается, даже в приложении с неустойчивыми
сегментами.

### Входные плейсхолдеры

| Плейсхолдер | Источник |
|---|---|
| `[КАТЕГОРИЯ]` | `cjm_<имя>.yaml: category` |
| `[БРЕНД/ПРЕПАРАТ]` | `cjm_<имя>.yaml: brand` |
| `[VERTICAL]` | `pharma_rx` \| `pharma_otc` |
| `[ТЕКСТ ИНСТРУКЦИИ]` | содержимое файла по пути `data_inputs.drug_instruction`, если задан |
| `[ИСТОЧНИК ИНСТРУКЦИИ]` | см. правило ниже — обязательное поле, не оставлять неявным |

**Правило источника инструкции (замена их «найди в интернете и используй»):**
если `data_inputs.drug_instruction` задан — источник это сам файл, указать путь.
Если не задан: `pharma_rx` без инструкции блокируется ещё на `cjm_init.py`
(отказ `REFUSAL_RX_NO_INSTRUCTION`, до этой стадии дело не доходит). Для
`pharma_otc` без инструкции — допустима ОДНА попытка найти официальную
инструкцию в открытых источниках (medi.ru, grls.rosminzdrav.ru и подобные) с
фиксацией URL и даты обращения; если попытка не удалась или веб недоступен —
использовать обобщённую типовую инструкцию категории с явной, прямо в
`00_filter.md` написанной пометкой «обобщённая/демо-инструкция категории, не
инструкция конкретного препарата — статус 🔴, не для реального клиента без
замены на настоящую инструкцию».

### Промпт

```
Ты выступаешь как эксперт в маркетинге фармацевтической отрасли с опытом в
анализе и сегментации фармацевтических целевых аудиторий, с учётом реалий
российского регулирования (ФЗ-38, ФЗ-61 и др.).

Проведи медицинский анализ препарата/категории [БРЕНД/ПРЕПАРАТ] ([КАТЕГОРИЯ]).

Источник инструкции: [ИСТОЧНИК ИНСТРУКЦИИ]. Используй именно этот текст:

---
[ТЕКСТ ИНСТРУКЦИИ]
---

Внимательно проанализируй и выдели:
- Показания к применению
- Противопоказания
- Возрастные ограничения
- Ограничения по беременности, лактации, хроническим состояниям
- Лекарственные взаимодействия

После анализа составь ФИЛЬТР ДОПУСТИМЫХ АУДИТОРИЙ в формате таблицы:

| Категория | Применение | Допустимо/Недопустимо | Комментарий |
|---|---|---|---|
| (пример) Дети до 18 лет | Противопоказано | Недопустимо | Указано в инструкции, п. ... |

В конце анализа чётко зафиксируй:
- Кому препарат/категория противопоказаны
- Кому не рекомендованы (мягкое ограничение, не жёсткое противопоказание —
  разводи эти два статуса явно, не смешивай в один)
- Кто входит в разрешённую целевую аудиторию для коммуникации

Явно выдели сегменты, которые должны быть ИСКЛЮЧЕНЫ из анализа на следующей
стадии (сегментации) — по названию/описанию группы, а не только по строке
таблицы, чтобы этим списком можно было руководствоваться на стадии 1.

[ЕСЛИ vertical = pharma_rx:] Дополнительно зафиксируй прямым текстом: препарат
рецептурный, коммуникация ограничена ФЗ-38 ст. 24 (только для специалистов
здравоохранения) — пациентские коммуникационные тесты на основе этого анализа
структурно ограничены и требуют отдельного юридического согласования до
использования вовне.

Не оценивай долю каждой категории аудитории в процентах и не присваивай
коэффициентов значимости — это фильтр допустимости (да/нет), не измерение
размера группы.
```

### Формат выхода

`00_filter.md` — таблица фильтра + три фиксации (противопоказан / не
рекомендован / разрешённая аудитория) + список исключаемых из стадии 1
сегментов + (для `pharma_rx`) предупреждение ФЗ-38 ст. 24.

`00_filter.yaml` — машиночитаемый компаньон (свободная схема, не потребляется
скриптами механически в v1.1, служит входом для стадии 1 и для сборки
`cjm_report.md`):

```yaml
category: "..."
brand: "..."
vertical: pharma_otc
instruction_source:
  status: предоставлена | найдена_веб | обобщённая_демо
  path_or_url: "..."           # путь к файлу ИЛИ URL
  accessed_at: "2026-07-09"    # только если status = найдена_веб
  trust: "🟢"                   # 🟢 предоставлена/найдена с URL; 🔴 обобщённая_демо
allowed_audience:
  - "..."
not_recommended:
  - "..."
excluded_segments:
  - "..."                       # НЕ создавать на стадии 1
rx_warning_included: true|false  # true только если vertical: pharma_rx
```

### Карта доверия

Сама таблица фильтра и фиксации — 🟢, если инструкция реальна (предоставлена
или найдена с URL); статус всего блока падает до 🔴, если инструкция
«обобщённая/демо» — это должно быть видно в `00_filter.md` открытым текстом, не
только в yaml-компаньоне.

## Стадия 1 — Сегментация

Переработка промптов 2–4 (поведенческо-мотивационный и альтернативные подходы
к сегментации) и промпта из «Промпт для анализа сегмента.docx». Сильное
сохранено: поведенческо-мотивационная рамка, JTBD, живой язык, чек-лист перед
сдачей. Убрано: «ровно 10 сегментов», проценты по упоминаемости, коэффициент
значимости 0.1–1.0, обещание привести примеры отзывов без указания источника.

### 1.0 (опционально) Разведка неосознанных потребностей

Кодификация «универсального промпта для поиска неосознанных потребностей»
(последний промпт первого файла источника, 5 аналитических каналов) — полезна
как ОБЩИЙ вход перед всеми тремя независимыми прогонами сегментации, когда
категория такова, что мотивация редко называется прямо (типичный случай для
здоровья/красоты). Это НЕ нарушает независимость прогонов 1.1 ниже: одинаковый
входной материал, переданный во все три прогона — это общая почва, а не
копирование сегментов друг у друга; независимость требуется именно от
результата (списка сегментов), не от входных данных.

```
Проведи разведку неосознанных, неартикулированных потребностей аудитории в
категории [КАТЕГОРИЯ] для бренда/препарата [БРЕНД] — потребностей, которые
существуют в жизни человека, но не формулируются им напрямую как запрос на
продукт.

Раскрой по пяти каналам (по 5–10 наблюдений в каждом, без процентов):

1. Живые пользовательские ситуации — в каких жизненных ситуациях потребность
   проявляется, но не называется напрямую (например: «волосы стали суше после
   отпуска» — не запрос на продукт, но боль есть).
2. Методологии скрытых мотиваций (JTBD, эмоциональные триггеры) — какую задачу
   человек реально решает для себя, даже не осознавая это явно.
3. Перенос из смежных категорий — как аналогичная боль решается в других
   сегментах рынка (wellness, косметика, питание) и что из этого можно
   легально перенести в [КАТЕГОРИЯ].
4. Контекст РФ, тренды, сезонность — если используешь факты из интернета,
   указывай URL и дату обращения в этом прогоне; без подтверждённого сейчас
   URL — помечай «по памяти модели, не проверено».
5. Гипотезы — сформулируй 1–2 гипотезы, как бренд может ответить на скрытую
   потребность этично и на языке пользователя, и что из этого стоит проверить
   на следующей стадии (сегментация/CJM).

Это разведка, не финальная сегментация — не подгоняй результат под конкретное
число сегментов.
```

Выход: `01_needs_exploration.md` (опционально; передаётся как общий контекст
во все три прогона 1.1, не заменяет их).

Карта доверия: 🟡 весь раздел целиком — это гипотезы для следующей стадии, не
факт.

### 1.1 Независимые прогоны сегментации (× 3, минимум)

**Изоляция контекста — обязательное условие, не стилистическая оговорка.** В
боевом режиме каждый из ≥3 прогонов выполняется в изолированном контексте
(отдельный субагент/отдельный вызов API, без общей истории с другими
прогонами) — в `manifest.json` фиксируется `independence: full`. В демо/пилоте
допустимо приближение: последовательные прогоны в ОДНОМ контексте с явной
инструкцией ниже — тогда в `manifest.json` фиксируется `independence:
approximated`, и это ограничение прямо указывается в `cjm_report.md` (шаблон —
`references/cjm_report_template.md`). Приближение — только для демо/пилота, не
для сдачи клиенту.

#### Входные плейсхолдеры

| Плейсхолдер | Источник |
|---|---|
| `[КАТЕГОРИЯ]`, `[БРЕНД]` | `cjm_<имя>.yaml` |
| `[ДОПУСТИМАЯ АУДИТОРИЯ]` | `00_filter.md` (только если стадия 0 пройдена; иначе — пусто) |
| `[ПОДХОД СЕГМЕНТАЦИИ]` | по умолчанию — поведенческо-мотивационный; можно явно задать один из 10 подходов их промпта 4 (по частоте симптома, по стадии осознания, по барьерам, по триггерам, по жизненной роли, по типу заботы о себе, по медиаповедению, по JTBD и т.д.) |
| `[ДАННЫЕ СОЦЛИСТЕНИНГА]` | путь `data_inputs.social_listening`, если задан; иначе явно «не передано» |
| `[РАЗВЕДКА НЕОСОЗНАННЫХ ПОТРЕБНОСТЕЙ]` | `01_needs_exploration.md`, если делали 1.0; иначе — пусто |
| `[N] / [M]` | номер этого прогона из общего числа (`runs_for_stability`) |
| `[ЦЕЛЕВОЙ ДИАПАЗОН]` | `segments_target_range`, напр. «от 5 до 12» |

#### Промпт

```
Ты — независимый аналитик, проводящий поведенческо-мотивационный анализ
аудитории категории [КАТЕГОРИЯ] для маркетинговой команды бренда/препарата
[БРЕНД]. Подход к сегментации: [ПОДХОД СЕГМЕНТАЦИИ].

ЭТО НЕЗАВИСИМЫЙ ПРОГОН [N] ИЗ [M]. Не опирайся на результаты других прогонов
этого же анализа, даже если они каким-то образом видны тебе в контексте —
игнорируй их формулировки сегментов и границы между сегментами. Не используй
готовые внешние списки сегментов категории/бренда как шаблон, к которому нужно
подогнать результат. Строй сегментацию заново, от категорийной логики и
переданных ниже материалов.

[ЕСЛИ фарма-гейт пройден:] Обязательный фильтр — работай ТОЛЬКО в рамках
разрешённой аудитории: [ДОПУСТИМАЯ АУДИТОРИЯ]. Не создавай и не упоминай
сегменты вне этого списка.

[ЕСЛИ есть 1.0:] Контекст для разведки скрытых потребностей (общий вход для
всех прогонов, не результат другого прогона сегментации):
[РАЗВЕДКА НЕОСОЗНАННЫХ ПОТРЕБНОСТЕЙ]

[ЕСЛИ есть соцлистенинг:] Материал для языка и иллюстраций (можно цитировать
дословно с указанием источника): [ДАННЫЕ СОЦЛИСТЕНИНГА]
[ИНАЧЕ:] Данных соцлистенинга не передано — иллюстративные формулировки
сегментов будешь придумывать сам, с обязательной пометкой «синтетическая
иллюстрация» у каждой (формулировки вида «отзыв пользователя» или «прямая
цитата» без указания источника здесь не используем).

Сформируй ОТ 5 ДО 12 уникальных сегментов ([ЦЕЛЕВОЙ ДИАПАЗОН] — это ориентир,
не жёсткая цель; финальное число решает устойчивость на следующем шаге
объединения прогонов, не подгоняй здесь искусственно). Исключи сегменты,
привязанные к разовым/устаревшим событиям, потерявшим актуальность
(«постковидные», «на фоне конкретного кризиса» и т.п.), если только это не
структурная, устойчивая черта категории.

Для каждого сегмента опиши:
- **Название** — описательное: кто это, в каком контексте находится.
- **Главная мотивация** — что хочет достичь; формулировка живым языком, не
  «улучшить здоровье/внешний вид» общими словами, а с контекстом и эмоцией:
  «хочу, чтобы Х перестало происходить», «мечтаю снова Y», «жду, что Z».
- **Главный барьер** — что мешает; тоже с эмоцией и контекстом, не «боязнь
  побочек» общими словами, а конкретный страх/сомнение своими словами сегмента.
- **JTBD-формулировка** — какую задачу человек реально решает для себя в этой
  категории (по методологии Jobs To Be Done), даже если сам не формулирует её
  так напрямую.
- **Матрица**: боль (эмоциональная/социальная/функциональная) → барьер →
  типичное поведение → к какому решению обычно склоняется.
- **Уникальные инсайты** — страхи, скрытые мотивы, характерные триггеры,
  которые отличают именно этот сегмент от других в этом же прогоне.
- **Психологический триггер** — что чувствует и как объясняет проблему своими
  словами.
- **Иллюстративные формулировки (3–5)** — живым языком сегмента. Если выше
  передан материал соцлистенинга — можно процитировать дословно оттуда, с
  указанием источника (например «[отзывы: путь к файлу], строка N» или как
  обозначено в самом файле). Если материала нет — формулировки твои
  собственные для иллюстрации характерной речи сегмента, и КАЖДАЯ подписана
  «синтетическая иллюстрация».
- **Оценка размера (не число)** — ранг относительно других сегментов ЭТОГО
  прогона: «вероятно один из крупных», «средний», «нишевый» — с пометкой
  «оценка, не измерение». Никаких процентов, никаких коэффициентов значимости
  0.1–1.0.

Если по ходу анализа используешь факт из интернета — указывай URL и дату
обращения В ЭТОМ прогоне; без подтверждённого сейчас URL — помечай «по памяти
модели, не проверено» и не выдавай за проверенный факт.

Чек-лист перед сдачей:
- Все сегменты — в рамках допустимой аудитории (если применим фарма-фильтр).
- Нет сегментов, привязанных к разовым устаревшим событиям.
- Ни одного процента и ни одного коэффициента значимости в тексте.
- Каждая иллюстративная формулировка помечена источником ИЛИ «синтетическая
  иллюстрация» — без исключений.
- Стиль — живой, практичный для маркетинга, как в исходном инструменте
  стратегов, а не сухой академический разбор.
```

#### Формат выхода

`01_segmentation_run{N}.md` (N = 1..`runs_for_stability`) — по одному блоку
на сегмент со всеми полями выше, в свободной markdown-форме (это черновой,
рабочий артефакт — не парсится скриптами, только читается на шаге объединения
ниже и агентом, пишущим `cjm_report.md`).

#### Карта доверия

Мотивация/барьер/JTBD/матрица/инсайты/триггер/язык — 🟢. Иллюстративные
формулировки — 🟢, если корректно подписаны (источник ИЛИ «синтетическая
иллюстрация»); отсутствие подписи — дефект черновика, чинится до объединения.
Оценка размера — 🔴 (ранг, не измерение).

### 1.2 Объединение прогонов → `01_segments_merged.yaml`

Это делает МОДЕЛЬ-ОРКЕСТРАТОР (не отдельный промпт-для-персоны) после того, как
получены все `01_segmentation_run{N}.md`. Правила объединения:

1. **Сопоставлять по мотивации + поведению, НЕ по названию.** Независимые
   прогоны почти никогда не дают одинаковых названий сегментов — это ожидаемо,
   не признак провала. Практический тест на совпадение: два сегмента из разных
   прогонов — один и тот же сегмент, если совпадают минимум 2 из 3: (а)
   главная мотивация относится к одному и тому же «job» (JTBD), (б) барьер
   того же типа (эмоциональный/рациональный/социальный, не обязательно
   дословно), (в) поведенческий паттерн совпадает по каналу/триггеру/типичному
   решению. Совпадение только по одному из трёх — недостаточно, это разные
   сегменты, даже если тема близкая.
2. **Stability-метка** — в скольких из `runs_for_stability` прогонов сегмент
   воспроизведён: `"3/3"`, `"2/3"`. Сегменты с меткой **≥2/3 идут в ядро**
   (`segments:`); сегменты, встретившиеся только в одном прогоне (`"1/3"`) —
   в `unstable_segments:` с пометкой «неустойчивый», НЕ в ядро.
3. **Название и описание объединённого сегмента** — синтез формулировок из
   всех совпавших прогонов (не механический выбор одного прогона наугад);
   зафиксировать, из каких прогонов собран сегмент (для `source`/для отчёта).
4. Если совпадающих сегментов между прогонами почти нет (каждый прогон дал
   полностью непересекающийся список) — это сигнал нестабильности категории
   для сегментации в принципе, фиксируется как явное наблюдение в
   `cjm_report.md`, не замалчивается и не «дожимается» искусственным
   притягиванием несовпадающих сегментов друг к другу.
5. Итоговое число сегментов ядра не подгоняется под `segments_target_range` —
   если после честного объединения ядро вышло за диапазон, это тоже
   фиксируется как наблюдение, а не повод придумать/слить сегменты, лишь бы
   попасть в диапазон.
6. Зафиксировать независимость прогонов в `manifest.json` (через
   `cjm_init.record_stage(run_dir, "segmentation", {...})` или прямой правкой
   JSON): `independence: full | approximated`, число прогонов, дата каждого.

самого прогона, не поле мнения, в карте доверия не участвуют как «модельное

### 2.1 Рамка «маркеры → симптомы → диагноз» (общий контекст, не отдельный файл)

Эта рамка — не промпт для отдельного прогона, а концептуальная опора,
включаемая ЦЕЛИКОМ как контекст в промпты 2.3–2.5 ниже (не пересказывать
своими словами — переносить структуру как есть, это прямая рекомендация
спецификации сохранить почти дословно):

```
При анализе этапа "Осознание проблематики" используй следующее разделение:

1. РАННИЕ МАРКЕРЫ ПРОБЛЕМЫ — незаметные или малозначимые признаки, ещё не
   интерпретируются человеком как проблема, находятся "на фоне", игнорируются.
   Не вербализуются как "у меня проблема" — это скорее "что-то странно",
   "показалось". Поведенческий отклик часто отсутствует.
2. ПОЗДНИЕ МАРКЕРЫ ПРОБЛЕМЫ — более заметные изменения, вызывающие осознание
   "что-то не так". Уже вербализуются: "у меня проблема с Х". Возникает
   тревога, поиск решения.
3. СИМПТОМЫ — осознанные нарушения нормы, часто с эмоциональной окраской. Это
   ключевой этап для запуска конверсии и триггерной коммуникации.
4. ДИАГНОЗ — официальное (врачебное) или самоопределённое признание проблемы.
   Переход к активным действиям и лечению.

Различие важно для тона коммуникации: при маркерах — мягкий, поддерживающий
тон; при симптомах и диагнозе — увереннный, предлагающий решение. Маркеры
важны для раннего захвата внимания, симптомы — ключ к конверсии.
```

### 2.2 Поправка на защитное поведение модели (обязательна для `vertical: pharma_*`, рекомендована для health/wellness категорий вне фармы)

```
ВАЖНАЯ ПОПРАВКА (не пропускать при анализе пациентских/потребительских тем
здоровья): модели систематически смещают симулированного пациента к более
«правильному», дисциплинированному поведению, чем реальные люди — это известное
ограничение, а не гипотеза (независимые исследования пациентских диалогов
фиксируют долю такого смещения в диапазоне нескольких десятков процентов
проблемных ответов — см. `Анализ_AI_CJM_и_интеграция_с_панелью.md`, п.6
таблицы слабостей). Явно компенсируй это:

- Включай нонкомплаенс как норму описания, не как исключение: пропуск приёма,
  самовольное изменение дозы/схемы, преждевременное прекращение при улучшении
  самочувствия, смешивание с БАД/народными средствами без консультации.
- Включай самолечение ДО обращения к врачу как основной, а не маргинальный
  путь: покупка без рецепта по совету фармацевта или отзывов, форумы вместо
  визита к специалисту — это типичное поведение в OTC-категориях, не
  отклонение от нормы, которое нужно замалчивать.
- Формулируй вопросы нейтральной рамкой: «как это обычно обсуждают на
  форумах/в отзывах», «что типично делают люди в такой ситуации» — а НЕ «что
  вы будете делать» или «правильно ли поступать так». Прямой вопрос о
  намерении провоцирует социально одобряемый ответ (к врачу, строго по
  инструкции), а не реалистичное поведение.
```

### 2.3 Промпт: Этап 1 — Осознание проблематики

#### Входные плейсхолдеры

`[СЕГМЕНТ]` (name + краткое описание из `01_segments_merged.yaml`),
`[КАТЕГОРИЯ]`, `[БРЕНД]`, `[ДОПУСТИМАЯ АУДИТОРИЯ]` (для фарма), рамка 2.1,
поправка 2.2 (для фарма/health).

```
Проанализируй этап "ОСОЗНАНИЕ ПРОБЛЕМАТИКИ" Customer Journey Map для сегмента
[СЕГМЕНТ] в категории [КАТЕГОРИЯ], для бренда/препарата [БРЕНД].

[Рамка маркеров/симптомов из 2.1]
[Поправка 2.2, если применимо]

Дай по каждому пункту живые формулировки — как сегмент говорит об этом своими
словами, с эмоциональной окраской. Если формулировка синтетическая (твоя, не
из переданных вербатимов) — помечай «синтетическая иллюстрация»; если материал
соцлистенинга был передан на стадии 1 — можешь опираться на реальные
формулировки оттуда с указанием источника.

1. Ранние маркеры (5-10 формулировок) — что человек может замечать, ещё не
   осознавая проблему.
2. Поздние маркеры и первые осознанные симптомы (5-10 формулировок) — что
   становится заметным и настораживающим.
3. Симптомы — детальный разбор (5-10 пунктов): какие проявления отмечает
   человек, КАКИМ ЯЗЫКОМ описывает (не частоту в процентах — просто какие
   формулировки типичны).
4. Самодиагностика и диагноз: какие диагнозы люди ставят себе сами; какими
   словами и терминами обычно оперирует врач при постановке диагноза для этой
   категории (в общих чертах, не изображая точную клиническую консультацию).
5. Соотношение "маркер / симптом / диагноз" в этом сегменте — БЕЗ ПРОЦЕНТОВ:
   ранговая оценка, каких историй в обсуждениях, судя по общей логике
   категории, вероятно больше — с явной пометкой 🔴 "оценка, не измерение". Если
   передана эпидемиологическая/категорийная статистика (данные стадии 3) —
   этот пункт будет переписан на стадии 3 с тегом источника вместо оценки.
6. Эмоциональная реакция — тревоги, страхи, злость, стыд, своими словами.
7. Убеждения о причинах — как сегмент объясняет себе, почему это случилось
   (версии, истории, поверья — не обязательно медицински точные).
8. Поведение после осознания — куда обращается в первые дни, с кем делится (с
   учётом поправки 2.2 — включая нонкомплаенс/самолечение как реалистичный
   вариант, не только "сразу к врачу").

Не используй процентные распределения нигде в этом разделе.
```

### 2.4 Промпт: Этап 2 — Поиск решения (выбор стратегии)

```
Продолжи CJM для сегмента [СЕГМЕНТ] — Этап 2: ПОИСК РЕШЕНИЯ.

[Поправка 2.2, если применимо]

1. Стратегии решения, которые рассматривает сегмент (самолечение, визит к
   врачу, консультация фармацевта, форумы/народные средства, игнорирование,
   диета/образ жизни) — какие в приоритете у ЭТОГО сегмента и почему, живым
   языком. Без процентов — ранжируй словами ("чаще всего... реже...").
   Упоминай прошлый опыт (были ли уже попытки решения) и нестандартные способы,
   которые рассматривались, но не были применены.
2. Факторы, влияющие на выбор стратегии — топ-5 для ЭТОГО сегмента (не в
   процентах, порядком значимости), и явно — что мешает выбрать активное
   лечение/решение (барьеры: страх, цена, недоверие, "само пройдёт", неудобство).
3. Источники информации и поведение — где ищет первичную информацию (не более
   8-10 каналов), как именно ищет в каждом (короткий сценарий), какие источники
   вызывают доверие, а какие нет, и почему.
4. Эмоции, сомнения, барьеры в момент поиска — 8-12 характерных фраз сегмента
   (с пометкой источника/«синтетическая иллюстрация»).
5. Поисковые запросы — 10-15 примеров типичных запросов в поисковике для этого
   сегмента и категории.
6. Критерии выбора решения/препарата — что важно (эффективность, скорость,
   безопасность, натуральность, цена, форма выпуска), что отталкивает.
7. Влияние других людей — РАНЖИРУЙ источники влияния от сильного к слабому
   ДЛЯ ЭТОГО СЕГМЕНТА словами (например: "мнение фармацевта > советы знакомых >
   реклама"), с коротким обоснованием порядка через паттерны поведения/цитаты —
   БЕЗ процентов на каждый источник. Если передан реальный опрос/соцлистенинг —
   веса можно указать с тегом источника вместо ранга.
8. Ожидания и опасения от решения/лечения — какого результата ждёт, какие
   страхи нужно снять сообщением.
9. Физиологические и регуляторные барьеры — ограничения по здоровью
   (аллергии/хронические состояния/возраст) и регуляторные моменты (рецепт,
   доступность, цена), если релевантно категории.

Никаких процентов и коэффициентов в этом разделе — только ранги и живой язык.
```

### 2.5 Промпт: Этап 3 — Процесс лечения/использования

```
Продолжи CJM для сегмента [СЕГМЕНТ] — Этап 3: ПРОЦЕСС ЛЕЧЕНИЯ/ИСПОЛЬЗОВАНИЯ.

[Поправка 2.2, если применимо]

- Как выбирает конкретное решение (препарат/врача/канал покупки)?
- Какие ожидания от результата? Что сегмент считает "эффективным решением" —
  5-10 формулировок своими словами.
- Следует ли инструкции/рекомендации или меняет схему сам (с учётом поправки
  2.2 — нонкомплаенс называем прямо, не приукрашиваем)?
- Что чувствует в процессе, как отслеживает эффект — 5-10 формулировок.
- Сравнивает ли с другими средствами/опытом?
- Делится ли обратной связью — где, с кем, в какой форме?

- Как меняет образ жизни/привычки после (профилактика, регулярный приём,

Без процентов. Формат — живые формулировки, как будто пересказ реальных

```

### 2.6 (опционально, рекомендуется перед стадией 4) Отличия от других сегментов ядра

Переработка промптов 8–9 — полезно как мост к RTB: если понятно, чем сегмент
принципиально отличается от соседних, легче сформулировать РАЗНЫЕ RTB-акценты
для теста на стадии 4.

```
Сравни сегмент [СЕГМЕНТ] с другими сегментами ядра этого CJM-прогона:
[СПИСОК ДРУГИХ СЕГМЕНТОВ ЯДРА].

По каждому пункту — не более 3-5 предложений, без процентов, только словесные
сравнения ("больше/меньше склонны", "чаще/реже", "в отличие от..."):
1. Чем стратегия решения проблемы у этого сегмента отличается от других?
2. Какие факторы выбора здесь значимее/менее значимы, чем у других?
3. Чем отличается эмоциональный фон и барьеры?
4. Какие каналы информации этот сегмент использует иначе?
5. Чем отличаются критерии выбора решения/препарата?
6. Кто (или что) сильнее влияет на выбор именно в этом сегменте?
```

## Стадия 4 — RTB-кандидаты и SSR-тест

Формула — почти дословно промпт 10 стратегов, с одной критической правкой:
**в оригинале пример RTB содержал вымышленную точность («клиническими
исследованиями на 1000+ пациентах») — это ровно тот тип фабрикации цифр,
который запрещён спецификацией.** Ниже — формула без придуманных чисел.

### 4.1 Формула RTB (промпт стратегов, адаптированная)

```
Для [сегмент, выбирающий такую-то стратегию решения] и испытывающих [главный
барьер/страх сегмента], [бренд/категория] предлагает [ключевой benefit —
обещание] благодаря [механизм действия, объяснённый простыми словами], что
подтверждено [RTB — тип доказательства].
```

**Правило для поля [RTB — тип доказательства]:** если в `data_inputs`
переданы реальные клинические/исследовательские данные — использовать их с
источником. Если нет (демо, вымышленный бренд, нет данных) — RTB строится
через МЕХАНИЗМ и ЛОГИКУ действия («благодаря составу, разработанному для...»,
«за счёт формулы, которая...»), либо через тип авторитета без придуманных
чисел («по рекомендации фармацевтов», «независимо от бренда — по принципу
действия компонента X»); придумывать конкретное число участников
исследования, конкретный процент эффективности или конкретную ссылку на
несуществующее исследование — запрещено категорически, это не «гипотеза для
проверки», а прямая фабрикация, неотличимая от их же примера с «1000+
пациентами».

### 4.2 Промпт: генерация N кандидатов на сегмент

#### Входные плейсхолдеры

`[СЕГМЕНТ]`, `[ЧИСЛО КАНДИДАТОВ]` (= `rtb_candidates_per_segment` из
`cjm_<имя>.yaml`, дефолт 4), `[РЕЗЮМЕ СТАДИИ 2.7]` (главные стратегии, топ-3
барьера, топ-2 benefit, механизм, доверенные источники), `[ДАННЫЕ СТАДИИ 3]`
(что реально подтверждено, если есть).

```
На основе резюме CJM сегмента [СЕГМЕНТ] (ниже) сформулируй [ЧИСЛО КАНДИДАТОВ]
РАЗНЫХ вариантов RTB-сообщения по формуле 4.1. Кандидаты должны различаться не
косметически (не перефразировка одного и того же), а по сути — так, чтобы
последующий тест панелью реально мог показать, какой сильнее для сегмента:

[РЕЗЮМЕ СТАДИИ 2.7]
[ДАННЫЕ СТАДИИ 3, если есть]

Варьируй между кандидатами минимум одно из:
- какой из топ-барьеров сегмента адресует сообщение (обычно 2-3 разных барьера
  дают заметно разные тексты);
- какой benefit выведен на первый план;
- тип доказательства/авторитета (механизм действия / рекомендация специалиста
  / натуральность состава / удобство использования) — без придуманных чисел
  (см. правило 4.1 про недопустимость выдуманной точности).

Каждый кандидат — 1-2 предложения, законченная фраза, естественный
маркетинговый русский язык, пригодная для показа респонденту как есть (это
будет стимул в SSR-тесте панели, не черновик для дальнейшей правки).

Для КАЖДОГО кандидата отдельно, помимо самого текста, зафиксируй ТРИ поля —
не растворяй их только в тексте кандидата, они нужны отдельно для таблицы §6.1
`cjm_report_template.md`:
- **targeted_barrier** — какой из топ-барьеров сегмента адресует этот кандидат
  (короткая формулировка, не весь барьер целиком).
- **targeted_benefit** — какой benefit выведен на первый план.
- **rtb_type** — тип доказательства/авторитета, использованный в этом
  кандидате (механизм действия / рекомендация специалиста / натуральность
  состава / удобство использования и т.п., без придуманных чисел — см. правило
  4.1 выше).

До теста панелью — статус каждого кандидата 🟡 "гипотеза для проверки", не
факт и не финальная рекомендация.
```

### 4.3 Формат выхода → вход для `scripts/cjm_to_study.py`

Файл `04_rtb_candidates.yaml` (или `.yml`/`.json`) в каталоге прогона — имя и
обе схемы зафиксированы в коде (`cjm_to_study.py`,
`RTB_CANDIDATES_FILENAMES`/`build_studies`). Ровно ОДНА из двух форм:

**`targeted_barrier`/`targeted_benefit`/`rtb_type` — ОБЯЗАТЕЛЬНЫ на каждого
кандидата** (находка №3, MAJOR, `docs/review_v1.1.md`): без них таблица §6.1
`cjm_report_template.md` («Целевой барьер» / «Целевой benefit» / «Тип RTB») не
из чего заполнить. `scripts/cjm_to_study.py` их не роняет и не требует
принудительно (старые `04_rtb_candidates.yaml` без этих полей по-прежнему
читаются — обратная совместимость, см. `normalize_candidates`/
`extract_rtb_metadata`), но НЕ записывает в `stimuli[]` study.yaml (схема v1
`run_study.py` — строго `{id, text}`, не меняется) — переносит их в `notes`
study.yaml как справочный текст для того, кто дальше собирает
`cjm_report.md`. Отсутствующее поле → «—» в соответствующей ячейке §6.1 при
сборке отчёта (обратная совместимость со старыми прогонами).

**Форма А — общий пул (рекомендуется по умолчанию)**, когда одни и те же
формулировки тестируются на нескольких сегментах — это и есть основная
ценность панели: сравнить реакцию разных сегментов на одинаковый текст:

```yaml
segments: [molodye_mamy_posle_rodov, drugoi_segment_id]   # id из ЯДРА 01_segments_merged.yaml
rtb_candidates:
  - id: rtb1
    text: "Для тех, кто выбирает..., и переживает из-за..., [бренд] предлагает... благодаря..."
    targeted_barrier: "..."     # ОБЯЗАТЕЛЬНО — барьер сегмента, который адресует кандидат
    targeted_benefit: "..."     # ОБЯЗАТЕЛЬНО — benefit, выведенный на первый план
    rtb_type: "..."             # ОБЯЗАТЕЛЬНО — тип доказательства/авторитета (см. правило 4.1)
  - id: rtb2
    text: "..."
    targeted_barrier: "..."
    targeted_benefit: "..."
    rtb_type: "..."
  # минимум 2 кандидата; ориентир по количеству — rtb_candidates_per_segment
```

**Форма Б — посегментно**, когда тексты намеренно разные для каждого сегмента
и кросс-тест одного текста на чужом сегменте не имеет смысла:

```yaml
by_segment:
  molodye_mamy_posle_rodov:
    - {id: rtb1, text: "...", targeted_barrier: "...", targeted_benefit: "...", rtb_type: "..."}
    - {id: rtb2, text: "...", targeted_barrier: "...", targeted_benefit: "...", rtb_type: "..."}
  drugoi_segment_id:
    - {id: rtb1, text: "...", targeted_barrier: "...", targeted_benefit: "...", rtb_type: "..."}
```

Каждый id тестируемого сегмента ДОЛЖЕН быть в ядре `01_segments_merged.yaml`
(`segments:`, не `unstable_segments:`) — иначе `cjm_to_study.py` откажет
(неустойчивый сегмент по умолчанию не тестируется RTB).

Дальше — механически, без нового промпта:

```
python scripts/cjm_to_study.py --run runs/cjm_<имя>_<ts>/
python scripts/run_study.py --study studies/<сгенерированный>.yaml --stage all
```

`run_study.py` в agent-режиме использует СУЩЕСТВУЮЩИЙ протокол генерации
ответов персон из `SKILL.md` §2 (2-5 предложений от первого лица, БЕЗ
рассуждений вслух, БЕЗ числовой оценки в тексте, вариативность между

### Промпт

#### Входные плейсхолдеры

Правила:
