## 1. Математика (`scripts/ssr_core.py` против §4 спецификации)

**Вывод: без критических замечаний.** Построчная сверка:

| Шаг формулы §4 | Реализация | Соответствие |
|---|---|---|
| `s_i = cosine(r, a_i)` | `cosine_similarity_matrix()` — L2-нормировка, включая защиту от деления на 0 у нулевых векторов | ОК |
| `p_i = (s_i − min_j s_j) + ε` | `pmf_from_similarities()`: `s_min = sim.min(axis=1)` (минимум ИМЕННО по 5 точкам шкалы для каждого ответа), `numerator = (sim − s_min) + epsilon` | ОК, ось минимума верная |
| нормировка `p_i / Σp` | `pmf = numerator / denom`, с явным откатом на равномерное распределение `1/n_points` при вырожденном случае (все sim в строке равны И ε=0 → denom=0) | ОК, задокументированный fallback, не NaN/crash |
| опциональное температурное сглаживание `softmax(log p / T)` | `apply_temperature()`; при T=1.0 — тождество (доказано в докстринге: `softmax(log p)=p`, т.к. Σp=1, и подтверждено тестом `test_temperature_one_is_identity`); T<1 заостряет, T>1 сглаживает (проверено тестом) | ОК |
| усреднение по наборам якорей (≥4) → по сэмплам → по респондентам | `average_pmfs()` (простое среднее, инвариантно к порядку — тест `test_average_pmfs_invariant_to_set_order` и `test_ssr_engine_score_invariant_to_anchor_set_list_order`) + `aggregate_pmfs_by_key()` (группировка+среднее, используется дважды: сэмплы→респондент, респонденты→сегмент) | ОК, порядок стадий усреднения соответствует §4 |
| `E = Σ i·p_i` | `expected_value()`, шкала по умолчанию `1..n_points` | ОК |
| CI — бутстреп по респондентам | `bootstrap_ci()`: точечная оценка = среднее исходных значений (не среднее по бутстреп-выборкам — методологически верно), percentile-бутстреп с `np.random.default_rng(seed)`, детерминирован при фиксированном seed (тест `test_same_seed_gives_identical_result`) | ОК |

Важное наблюдение по корректности бутстрепа: докстринг обосновывает, что бутстрепить
можно скалярные E-значения на респондента вместо полных PMF-векторов, потому что
`E(mean(pmf_i)) == mean(E(pmf_i))` (E — линейный функционал). Проверил это утверждение
отдельно — оно математически верно и означает, что для КАЖДОЙ конкретной бутстреп-выборки
индексов результат идентичен тому, что дало бы усреднение PMF с последующим взятием E.
Упрощение корректно, не искажает CI.

**Юнит-тесты:** запустил `python3 scripts/test_ssr.py` — **23/23 зелёных**, без сети и без
`sentence_transformers`/`torch` (модуль не импортирует их на уровне файла, только лениво
внутри `SentenceTransformerBackend.__init__`, которую тесты не вызывают — подтверждено
чтением кода и успешным прогоном). Независимо подтверждает цифру «23/23», заявленную
сборщиком [B2] в его записке.

**Собственная интеграционная проверка (доп. к юнит-тестам):** т.к. `run_study.py`
отсутствует (см. находку №1), написал временный скрипт вне зоны проекта
(`scratchpad/integration_probe.py`), который берёт РЕАЛЬНЫЕ файлы репозитория
(`studies/demo_claims.yaml`, `panel/segments/{dessertnye,eksperty_kofemany}.yaml`,
`references/anchors_ru.yaml`) и прогоняет их через реальные `generate.py`/`ssr_core.py`
до стадии score (embedding-бэкенд — детерминированный SHA256-хэш вместо реальной модели,
чтобб не тянуть сеть/torch специально для pipeline-теста). Результат: 300/300 задач

бутстреп-CI посчитался по всем 10 парам (сегмент, стимул). **Вывод: код, который

притягивает»), а не как часть промпта генерации ответа персоны. Это не утечка.

Единственный **остаточный риск** (см. находку №5): в agent-режиме модель, ведущая
скилл, к моменту генерации ответов персон уже могла прочитать `anchors_ru.yaml`
(например, чтобы выбрать `question_scale` для `study.yaml`) — то есть защита здесь
инструкционная («не подглядывать»), а не структурная, в отличие от api-режимов, где
`generate.py` в принципе не может получить доступ к фразам якорей. SKILL.md уже
формулирует это предупреждение явно (раздел «Протокол agent-режима», пункт «Не
подглядывать в anchors_ru.yaml»), так что это не новая дыра, а известное и настолько,
насколько возможно, смягчённое архитектурное ограничение — фиксирую его отдельной
строкой, чтобы не считалось «полностью решённым».

## 3. Якоря (`references/anchors_ru.yaml`)

3 шкалы (`purchase_intent`, `appeal`, `relevance`) × 4 набора × 5 фраз = 12 наборов,
проверил каждый вручную:

- **Монотонность (1<2<3<4<5):** подтверждена во всех 12 наборах без исключений —
  прочитал и мысленно переставил силу каждой фразы i против i-1 в каждом наборе.
- **Естественность русского:** разговорные, живые формулировки («даже даром не нужно»,
  «как-то не моё», «оторваться просто не могу»); калек с английских reference-статей не
  нашёл.
- **Доменная независимость:** ни брендов, ни категорий, ни продуктов ни в одной из 60
  фраз — переиспользуемы для любого стимула, как и требуется.
- **≥4 набора на шкалу:** выполнено ровно по минимуму (4, не больше) на все 3 шкалы —
  требованию соответствует, но запаса сверх минимума нет (в тех-досье `06_...md`
  упоминается «4-6 наборов» как практика — не дефект, просто отмечаю отсутствие запаса).
- **Лексическое разнообразие внутри шкалы:** подтверждено — у каждой шкалы 4 набора
  осознанно используют разные регистры (например, для `purchase_intent`: прямое решение
  / желание-тяга / рассмотрение-вероятность / эмоциональный отклик), это явно
  задокументировано в поле `comment` каждой шкалы и фактически выполнено в тексте фраз.
- **Гендерная безопасность:** в преамбуле файла явно запрещены гендерно-маркированные
  формы («купил/купила», «готов/готова»). Проверил все 60 фраз на прошедшее время
  1-го лица и краткие прилагательные — нарушений не нашёл; там, где встречаются формы
  вроде «зацепило», «появилось», «впечатление осталось», грамматический субъект — сам
  стимул/впечатление (безличная конструкция), а не говорящий, что и требовалось.
