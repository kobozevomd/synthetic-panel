## Вердикт

**Основной пайплайн (`cjm_init.py` → `segments_export.py`/`cjm_to_study.py` → `run_study.py`)
работает надёжно и отказоустойчиво** — все проверенные ветки валидации (фарма-гейт,
минимум устойчивости, неверный тип, неустойчивый сегмент как RTB-цель, обе формы RTB
одновременно, дубликат id кандидата, <2 кандидатов) отказывают ровно так, как
задокументировано, с полезным сообщением и exit 1. Активы цепочки стратегов (фильтр по
инструкции, рамка маркеров/симптомов/диагноза, JTBD-каналы, формула RTB, медиашаблон)
сохранены добросовестно, слабости (проценты, «ровно 10», сочинённые цитаты) вычищены из
промпт-шаблонов без рецидива.

**Но линтер честности (`cjm_lint.py`) — механизм, на который явно опирается DoD §5.2
(«линтер — exit 0») — можно обмануть систематически, а не только в уже известном
частном случае.** Нашёл новый класс находок (табличное «отмывание» непомеченных
процентов/цитат через соседние строки одного блока — находка №1, CRITICAL) и уточнил уже
известную находку по маскировке CJM-дисклеймера (находка №2, CRITICAL): предложенный ранее
однострочный фикс необходим, но **недостаточен** — после его применения линтер всё равно
падает на том же самом отчёте по ДРУГОЙ причине. Обе находки воспроизведены полными
рабочими фикстурами ниже. Дополнительно — предметный пробел в схеме между
`cjm_report_template.md` (ожидает поля «целевой барьер/benefit/тип RTB» на кандидата) и
зафиксированной в коде схемой `04_rtb_candidates.yaml` (только `id`+`text`) — находка №3,
MAJOR.

### 3.2 Находка №1 (CRITICAL) — табличное «отмывание» непомеченных значений

`split_into_blocks()` в `cjm_lint.py` объединяет подряд идущие непустые строки одного
абзаца/пункта в один логический блок — это осознанное и правильное решение против
ложных срабатываний на переносах прозы (документировано в docstring). Но правило
распространяется на ЛЮБОЙ ряд непустых строк без разделителя-пустой строки, включая
строки markdown-таблицы (таблица физически не может иметь пустых строк между рядами, не
разрушив рендеринг) — а таблицы это ровно то, чем являются секции 3/5/6.1/6.2
`cjm_report_template.md`.

Проверки `check_percent_sourcing`/`check_real_quote_sourcing`/`check_forbidden_promises`
спрашивают «есть ли тег/маркер ГДЕ-ТО в тексте блока», а не «есть ли он рядом именно с
ЭТИМ процентом/цитатой». Значит один корректно помеченный ряд таблицы освобождает от
проверки ВСЕ остальные ряды той же таблицы. Воспроизвёл для всех трёх правил:

```python
# Правило 1 — 2 ряда одной таблицы, второй БЕЗ тега и БЕЗ слова оценка/гипотеза
"""
| Сегмент | Доля | Источник |
|---|---|---|
| Альфа | 42% | [BA] |
| Бета | 55% | без источника здесь вообще |
"""
# -> rule1 violations: 0   (ожидалось: 1, на ряду "Бета")

# Правило 2 — второй ряд содержит "Реальный отзыв" без тега
"""
| Сегмент | Цитата |
|---|---|
| Альфа | Реальный отзыв [BA]: "работает" |
| Бета | Реальный отзыв: "не работает вообще, зря деньги" |
"""
# -> rule2 violations: 0   (ожидалось: 1)

# Правило 4 (формулировка точности) — второй ряд с плохой формулировкой рядом с канонической
"""
| Сегмент | Точность |
|---|---|
| Альфа | Точность метода 90% для этого сегмента отдельно |
| Бета | Метод SSR: R=0,72 (~90% теоретического потолка) |
"""
# -> rule4 violations: 0   (ожидалось: 1, на ряду "Альфа")
```

Контрольный тест — та же пара строк из первого примера, но с пустой строкой между ними
(→ два ОТДЕЛЬНЫХ блока вместо одного): линтер корректно находит 1 нарушение на строке
"Бета". Это доказывает, что причина именно в объединении строк без разделителя, а не в
чём-то другом.

Это не надуманный edge-case: секции 3 («Сегменты: сводная таблица»), 5 («Данные и
источники»), 6.1 («RTB-кандидаты») и 6.2 («Результат SSR-теста») `cjm_report_template.md`
— ВСЕ многострочные markdown-таблицы. Модель, честно протегировавшая 3 из 4 строк
таблицы данных и один раз забывшая тег/слово-оценку на 4-й — ровно тот сценарий, для
которого линтер и создавался — пройдёт линтер незамеченной, если в той же таблице есть
хотя бы одна корректная строка. `test_cjm_lint.py` этот случай не покрывает ни одной
фикстурой (все примеры — однострочные или однопараграфные, ни одной таблицы).

**Рекомендация (не мой мандат чинить):** для правил 1/2/4 проверять тег/маркер на
уровне СТРОКИ ТАБЛИЦЫ (разбивать блок на под-единицы по `|`-рядам, если блок оказался
таблицей), а не всего merged-блока целиком; либо, минимально, детектировать таблицы
(`^\s*\|`) как отдельный тип блока, где КАЖДЫЙ ряд — своя единица проверки, в отличие от
прозы, где склейка ряда оправдана.

### 3.4 Находка №4 (MAJOR) — процент без цифр

```python
text = "Сорок процентов сегмента предпочитают этот вариант, без каких-либо оговорок."
# PERCENT_RE = r"\d\s?%" не матчит вообще -> rule1 violations: 0
```

`PERCENT_RE` требует буквальную цифру перед «%». Числительное словом («сорок», «треть»,
«почти половина») или голое слово «доля»/«процент» без символа `%` полностью обходит
правило 1, хотя по духу спецификации (§1.2: «LLM не генерирует проценты») это то же самое
нарушение. Не блокирует текущий DoD (в примерах шаблона везде цифровая форма), но это
дыра, которой явно не хватает отдельной фикстуры/правила (например: искать также
числительные словами рядом со словом «процент»/«доля» — сложнее регэкспом, но
осуществимо, или явно зафиксировать как «известное ограничение» в docstring, как уже
сделано для пункта про правило 2).

### 3.5 Проверенные и отклонённые гипотезы обмана

- **`%%` (двойной знак процента):** не обходит — `re.search(r"\d\s?%", "42%%")` всё равно
  находит `2%` как подстроку. Не находка.
- **Слово «гипотетически» вместо «гипотеза»:** НЕ обходит правило — `ESTIMATE_WORD_RE`
  (`\bоцен\w*|\bгипотез\w*`) корректно НЕ матчит «гипотетически» (разные корни:
  «гипотет-ически» ≠ «гипотез-»), поэтому «Гипотетически, около 30% аудитории...» ловится
  как нарушение (проверил фикстурой: 1 нарушение, как и должно быть). Это правильное,
  консервативное поведение линтера — я ошибочно ожидал бегства через это слово, но
  проверка показала обратное. Фиксирую как подтверждённый НЕ-баг, чтобы не тратить время
  следующего проверяющего на ту же гипотезу.
